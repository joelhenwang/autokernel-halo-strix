**Date:** 2026-05-07

## 1. Goal

Reduce NorMuon's measured 17.8% throughput cost (Run 2, full recipe) toward
the original Sprint 1 gate of ≤7% while keeping quality (wikitext_val BPB,
final training loss) within 1% of Run 2's observed values. User directive
is explicit: *optimize aggressively, minimize quality drop.*

**Starting baseline (Run 2, commit 511d675 era):**
- Final loss 4.4736 (vs AdamW baseline 4.7975 = −6.8%)
- wikitext_val BPB 1.893 (vs baseline 1.9214 = −1.5%)
- Throughput 32,478 tok/s (vs baseline 39,538 = 17.8% cost)
- Memory 10.1 GB (baseline 10.3 GB)

**Target envelope:**
- Throughput cost: best achievable ≤1% quality regression
- Quality floor: final loss ≤ 4.518 (within 1% of 4.4736)
- Quality floor: wikitext_val BPB ≤ 1.912 (within 1% of 1.893)

## 2. Roadmap context

Sprint 1 (2026-05-06, commits afd7dcb + 75cd918 + 511d675) delivered
NorMuon + architectural additions but exceeded the throughput gate. This
sub-sprint rectifies NorMuon's throughput cost so the full Sprint 1 recipe
becomes the justified default for Sprint 3's dolma-10B run.

**Value proposition:** every percentage-point of throughput saved here
amortizes over the ~50-hour Sprint 3 training run. Reducing NorMuon's
cost from 17.8% to 10% saves ~4 hours of dolma training.

## 3. Scope

### In scope

A five-phase measurement-driven descent. Each phase has a decision gate:
if target is met, stop descending. If quality regresses, revert the
offending change and continue without it.

| Phase | Activity | Est. compute | Est. dev time |
|---|---|---|---|
| A | Profile NorMuon step (three configs) | 15 min | 1 hr |
| A.5 | Analyze torch.compile graph breaks + NS micro-benchmark | 5 min | 30 min |
| B | Quick-win ablations (fp16 NS, size-gated neuron-norm, no-cautious-WD, combo) | 60 min | 2 hr |
| C | Structural optimizations (batched NS, reduced iterations) | 1 hr | 4 hr |
| D | Custom HIP kernel for Newton-Schulz (only if B + C insufficient) | 30 min | 6-12 hr |
| E | Final full-epoch validation with best config | 60 min | 30 min |

**Total worst case:** ~20 hours dev, ~4 hours compute.
**Total expected (stop at Phase B or C):** ~6-8 hours dev, ~2 hours compute.

### Out of scope

- Algorithm rewrite beyond the knobs available in `halo_training/normuon.py`
- Changes to value_residuals / head_gating implementation (already added
  only 2 pp cost above Probe C; marginal; can revisit if Phase D overruns)
- Non-NorMuon optimizations (e.g., further block/batch tuning) — defer to
  a separate sweep once NorMuon is in a good place
- New dataset pretokenization or model-architecture changes

### Non-goals

Even if Phase B hits the target, we still run Phase A profiling first so
the findings become durable knowledge for future optimizer work. Do NOT
skip Phase A.

## 6. Phase C — Structural Optimizations

### 6.1 C1: Batched Newton-Schulz

**Motivation:** OdinFlat-122M has 12 identical HyPEShortConv SwiGLU inputs,
12 identical SwiGLU outputs, 12 identical gate matrices. Each is NS'd
independently as 5 × 3 matmul sequences today. Stacking same-shape
parameters into a 3D tensor and running NS on the batch is algorithmically
equivalent but runs the inner matmuls as batched ops (one kernel launch
instead of 12).

**Implementation sketch:**

```python
# halo_training/normuon.py

from collections import defaultdict

def _normuon_step(self, group):
    ...
    # Build shape -> [params] map
    shape_groups = defaultdict(list)
    params_no_grad = []
    for p in group["params"]:
        if p.grad is None or p.ndim != 2:
            params_no_grad.append(p)
            continue
        shape_groups[tuple(p.shape)].append(p)

    for shape, plist in shape_groups.items():
        if len(plist) == 1:
            self._single_param_update(plist[0], group)
        else:
            self._batched_update(plist, group)  # stack + batched NS
```

`_batched_update` stacks gradients / momentum into `[K, M, N]` tensors,
runs `_batched_newton_schulz` (which operates on batched dims via `@`),
then un-stacks to update individual params.

**Batched Newton-Schulz:** the iteration `X = a*X + (b*A + c*A@A) @ X`
already works for batched `X` of shape `[K, M, N]` if `A = X @ X.transpose(-2, -1)`
has shape `[K, M, M]`. Normalization step needs per-batch norm:
`X = X / (X.norm(dim=(-2, -1), keepdim=True) * 1.02 + 1e-7)`.

**Correctness test:** `scripts/test_sprint1_1_normuon.py::test_batched_ns_matches_sequential`
— generate 8 random 2D matrices, run both sequential and batched NS,
assert outputs match within 1e-5.

**Expected gain:** 12 matrices × 5 iters × 3 matmuls = 180 matmul launches
collapses to 5 × 3 = 15 batched matmul launches. Each batched launch
is 12× the size but near-ideal utilization on GPU. Net 3-5× speedup
on these groups.

### 6.2 C2: Reduced NS iterations

Compute-cheap, quality-risk. NS iterations are linear in cost; reducing
5 → 3 gives 1.67× speedup on NS step. Orthogonality error grows from
~0.08 to ~0.25.

Test as a separate ablation after C1: if C1 meets target, don't bother
with C2. If not, run C2 on top of C1.

| Run | Change from C1 winner | Expected |
|---|---|---|
| C2a | `ns_steps=4` | 1.25× NS speed, err ~0.12 |
| C2b | `ns_steps=3` | 1.67× NS speed, err ~0.25 |

## 7. Phase D — HIP Newton-Schulz Kernel (conditional)

Only executed if Phase C does not suffice. Writes a fused HIP kernel for
the 5-iteration NS path.

### 7.1 Design

- File: `kernels/hip/fused_newton_schulz.py` (NEW)
- Follows the pattern of `kernels/hip/fused_rope_gate_mul.py`
- Single-kernel impl that takes `[K, M, N]` fp16 stacked gradients and
  returns `[K, M, N]` orthogonalized fp16 tensor
- Uses half2 loads for throughput; fp32 accumulators for numerical safety
- ns_steps as compile-time or runtime parameter (start with compile-time
  for simplicity; support 3 and 5 via two specialized kernels if needed)

### 7.2 Validation

- `scripts/bench_newton_schulz.py` extends to include HIP kernel
- `scripts/test_sprint1_1_normuon.py::test_hip_ns_matches_reference`
  — HIP output within 1e-4 of PyTorch reference for random inputs
- Rollback: fall back to PyTorch C1-batched path if HIP unavailable

### 7.3 Gate D → E

Even if the HIP kernel works, apply it ONLY via an opt-in CLI flag
(`--ns-backend {pytorch, hip}`, default `pytorch`) so fallback is trivial.

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|:---:|:---:|---|
| fp16 NS produces non-orthogonal output | Low | Medium | Micro-benchmark verifies; fallback to fp32 |
| Batched NS has different convergence than sequential | Low | Medium | Unit test asserts equivalence |
| Reduced ns_steps hurts quality more than expected | Medium | Medium | Ablate in isolation; revert if BPB regresses |
| HIP kernel has correctness bugs | Medium | High | Extensive validation; kernel is opt-in via flag |
| Phase A profile inconclusive (no clear bottleneck) | Low | Medium | Proceed to Phase B regardless; all B knobs have independent value |
